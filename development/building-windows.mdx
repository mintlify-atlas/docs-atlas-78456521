## Repository

Clone the [Zed repository](https://github.com/zed-industries/zed):

```bash
git clone https://github.com/zed-industries/zed.git
cd zed
```

## Dependencies

<Steps>
  <Step title="Install Rust">
    Install [rustup](https://www.rust-lang.org/tools/install).
  </Step>

  <Step title="Install Visual Studio or Build Tools">
    Choose one of the following:

    **Option A: Visual Studio**

    Install [Visual Studio](https://visualstudio.microsoft.com/downloads/) with these optional components:
    - `MSVC v*** - VS YYYY C++ x64/x86 build tools`
    - `MSVC v*** - VS YYYY C++ x64/x86 Spectre-mitigated libs (latest)`

    (Where `v***` is your VS version and `YYYY` is the release year. Adjust architecture as needed.)

    **Option B: Build Tools only**

    Install only the [Build Tools](https://visualstudio.microsoft.com/visual-cpp-build-tools/) plus the libs above and the "Desktop development with C++" workload.

    <Warning>
    This setup is not picked up automatically by rustup. Before compiling, initialize environment variables by launching the developer shell (cmd/PowerShell) installed in the Start menu or Windows Terminal.
    </Warning>
  </Step>

  <Step title="Install Windows SDK">
    Install the Windows 11 or 10 SDK for your system, ensuring at least `Windows 10 SDK version 2104 (10.0.20348.0)` is installed.

    Download from the [Windows SDK Archive](https://developer.microsoft.com/windows/downloads/windows-sdk/).
  </Step>

  <Step title="Install CMake">
    Install [CMake](https://cmake.org/download) (required by [a dependency](https://docs.rs/wasmtime-c-api-impl/latest/wasmtime_c_api/)).

    Or install it through Visual Studio Installer, then manually add the `bin` directory to your `PATH`, for example:

    ```
    C:\Program Files\Microsoft Visual Studio\2022\Community\Common7\IDE\CommonExtensions\Microsoft\CMake\CMake\bin
    ```
  </Step>
</Steps>

### PostgreSQL configuration notes

Update `pg_hba.conf` in the `data` directory to use `trust` instead of `scram-sha-256` for the `host` method. Otherwise, the connection fails with `password authentication failed`.

The file is typically at `C:\Program Files\PostgreSQL\17\data\pg_hba.conf`. After the change:

```conf
# IPv4 local connections:
host    all             all             127.0.0.1/32            trust
# IPv6 local connections:
host    all             all             ::1/128                 trust
```

If you are using a non-Latin Windows locale, set the `lc_messages` parameter in `postgresql.conf` (in the `data` directory) to `English_United States.1252` (or another UTF-8-compatible encoding):

```conf
# lc_messages = 'Chinese (Simplified)_China.936'
lc_messages = 'English_United States.1252'
```

After this, restart the `postgresql` service:
1. Press `Win`+`R` to open the Run dialog
2. Enter `services.msc` and select OK
3. Find `postgresql-x64-XX`, right-click it, and select Restart

## Building from source

Once you have the dependencies installed, you can build Zed using [Cargo](https://doc.rust-lang.org/cargo/).

<Tabs>
  <Tab title="Debug build">
    ```bash
    cargo run
    ```

    Debug builds are unoptimized and include debugging symbols. Use these for development and debugging.
  </Tab>

  <Tab title="Release build">
    ```bash
    cargo run --release
    ```

    Release builds are optimized and suitable for performance testing.
  </Tab>
</Tabs>

### Running tests

```bash
cargo test --workspace
```

<Note>
Visual regression tests are currently macOS-only and require Screen Recording permission. See [Building Zed for macOS](/development/building-macos#visual-regression-tests) for details.
</Note>

## Installing from msys2

Zed does not support unofficial MSYS2 Zed packages built for Mingw-w64. Please report any issues you may have with [mingw-w64-zed](https://packages.msys2.org/base/mingw-w64-zed) to [msys2/MINGW-packages/issues](https://github.com/msys2/MINGW-packages/issues?q=is%3Aissue+is%3Aopen+zed).

Please refer to [MSYS2 documentation](https://www.msys2.org/docs/ides-editors/#zed) first.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Setting RUSTFLAGS env var breaks builds">
    If you set the `RUSTFLAGS` env var, it will override the `rustflags` settings in `.cargo/config.toml` which is required to properly build Zed.

    Because these settings change over time, the resulting build errors may vary from linker failures to other hard-to-diagnose errors.

    If you need extra Rust flags, use one of the following approaches in `.cargo/config.toml`:

    **Add flags in the build section:**

    ```toml
    [build]
    rustflags = ["-C", "symbol-mangling-version=v0", "--cfg", "tokio_unstable"]
    ```

    **Add flags in the windows target section:**

    ```toml
    [target.'cfg(target_os = "windows")']
    rustflags = [
        "--cfg",
        "windows_slim_errors",
        "-C",
        "target-feature=+crt-static",
    ]
    ```

    **Or create a new `.cargo/config.toml` in the parent directory:**

    This is useful in CI because you do not need to edit the repo's original `.cargo/config.toml`.

    ```
    upper_dir
    ├── .cargo          # Make this folder
    │   └── config.toml # Make this file
    └── zed
        ├── .cargo
        │   └── config.toml
        └── crates
            ├── assistant
            └── ...
    ```

    In the new (above) `.cargo/config.toml`, if we wanted to add `--cfg gles`, it would look like:

    ```toml
    [target.'cfg(all())']
    rustflags = ["--cfg", "gles"]
    ```
  </Accordion>

  <Accordion title="Cargo errors claiming dependency uses unstable features">
    Try cleaning and rebuilding:

    ```bash
    cargo clean
    cargo build
    ```
  </Accordion>

  <Accordion title="STATUS_ACCESS_VIOLATION">
    This error can happen if you are using the "rust-lld.exe" linker. Consider trying a different linker.

    If you are using a global config, consider moving the Zed repository to a nested directory and add a `.cargo/config.toml` with a custom linker config in the parent directory.

    See [issue #12041](https://github.com/zed-industries/zed/issues/12041) for more information.
  </Accordion>

  <Accordion title="Invalid RC path selected">
    Sometimes, depending on security rules applied to your laptop, you may get:

    ```
    error: failed to run custom build command for `zed(C:\Users\USER\src\zed\crates\zed)`

    Caused by:
      process didn't exit successfully: `C:\Users\USER\src\zed\target\debug\build\zed-b24f1e9300107efc\build-script-build` (exit code: 1)
      --- stdout
      cargo:rerun-if-changed=../../.git/logs/HEAD
      cargo:rustc-env=ZED_COMMIT_SHA=25e2e9c6727ba9b77415588cfa11fd969612adb7
      cargo:rustc-link-arg=/stack:8388608
      cargo:rerun-if-changed=resources/windows/app-icon.ico
      package.metadata.winresource does not exist
      Selected RC path: 'bin\x64\rc.exe'

      --- stderr
      The system cannot find the path specified. (os error 3)
    warning: build failed, waiting for other jobs to finish...
    ```

    To fix this, manually set the `ZED_RC_TOOLKIT_PATH` environment variable to the RC toolkit path. Usually:

    ```
    C:\Program Files (x86)\Windows Kits\10\bin\<SDK_version>\x64
    ```

    See [issue #18393](https://github.com/zed-industries/zed/issues/18393) for more information.
  </Accordion>

  <Accordion title="Build fails: Path too long">
    You may receive an error like:

    ```
    error: failed to get `pet` as a dependency of package `languages v0.1.0`

    Caused by:
      failed to load source for dependency `pet`

    Caused by:
      Unable to update https://github.com/microsoft/python-environment-tools.git

    Caused by:
      path too long: 'C:/Users/runneradmin/.cargo/git/checkouts/python-environment-tools-903993894b37a7d2/ffcbf3f/crates/pet-conda/tests/unix/conda_env_without_manager_but_found_in_history/some_other_location/conda_install/conda-meta/python-fastjsonschema-2.16.2-py310hca03da5_0.json'
      class=Filesystem (30)
    ```

    To fix this, enable long-path support for both Git and Windows.

    **For Git:**

    ```bash
    git config --system core.longpaths true
    ```

    **For Windows (PowerShell):**

    ```powershell
    New-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\FileSystem" -Name "LongPathsEnabled" -Value 1 -PropertyType DWORD -Force
    ```

    See [win32 docs](https://learn.microsoft.com/en-us/windows/win32/fileio/maximum-file-path-limitation?tabs=powershell) for more information.

    <Warning>
    You need to restart your system after enabling long-path support.
    </Warning>
  </Accordion>

  <Accordion title="Graphics issues">
    ### Zed fails to launch

    Zed currently uses Vulkan as its graphics API on Windows. If Zed fails to launch, Vulkan is a common cause.

    You can check the Zed log at:

    ```
    C:\Users\YOU\AppData\Local\Zed\logs\Zed.log
    ```

    If you see messages like:
    - `Zed failed to open a window: NoSupportedDeviceFound`
    - `ERROR_INITIALIZATION_FAILED`
    - `GPU Crashed`
    - `ERROR_SURFACE_LOST_KHR`

    Vulkan may not be working correctly on your system. Updating GPU drivers often resolves this.

    If there's nothing Vulkan-related in the logs and you happen to have Bandicam installed, try uninstalling it. Zed is currently not compatible with Bandicam.
  </Accordion>
</AccordionGroup>
