Zed is organized as a Cargo workspace with over 200 crates, each serving a
specific purpose in the application architecture. This modular structure enables
code reuse, clear dependency boundaries, and parallel compilation.

## Workspace Organization

The workspace is defined in the root `Cargo.toml` and includes three main
categories:

### Core Application Crates

Crates in `crates/` that make up the main Zed application:

- **zed**: The main application binary and entry point
- **editor**: Text editing functionality and the `Editor` entity
- **workspace**: Root UI container managing panes, panels, and docks
- **project**: Project-level operations, LSP integration, and file management
- **worktree**: File tree representation (local or remote)
- **language**: Language definitions, syntax trees, and parsing
- **lsp**: Language Server Protocol client implementation

### Extensions

Crates in `extensions/` that provide additional language support:

- `extensions/glsl`: GLSL language support
- `extensions/html`: HTML language support
- `extensions/proto`: Protocol Buffer support
- `extensions/slash-commands-example`: Example slash command extension
- `extensions/test-extension`: Testing extension framework

### Tooling

Crates in `tooling/` for development and build processes:

- `tooling/xtask`: Build automation and development tasks
- `tooling/perf`: Performance benchmarking and profiling

## Dependency Patterns

### Foundation Layer

Low-level crates with minimal dependencies:

- `collections`: Custom collection types (HashMap, Vec alternatives)
- `rope`: Rope data structure for text
- `sum_tree`: B-tree variant for ordered sequences
- `util`: General utilities
- `clock`: Lamport clock for distributed systems

### Core Layer

Crates that build on foundation:

- `text`: Text representation using `rope`
- `gpui`: UI framework using foundation crates
- `rpc`: Communication protocol
- `proto`: Protocol definitions

### Feature Layer

Crates implementing specific features:

- `editor`: Depends on `text`, `language`, `project`, `gpui`
- `project`: Depends on `worktree`, `language`, `lsp`, `fs`
- `workspace`: Depends on `editor`, `project`, `gpui`

### Application Layer

Top-level crates:

- `zed`: Main application, depends on most feature crates
- `collab`: Server application, shares protocol crates with client

## Naming Conventions

Zed follows specific naming patterns:

- **Library path**: Most crates use `src/<crate_name>.rs` instead of `src/lib.rs`
  - Example: `editor` crate uses `src/editor.rs`
  - Specified in `Cargo.toml` as `[lib] path = "src/editor.rs"`
- **Store suffix**: Crates with `Store` suffix abstract over local/remote
  operations
  - Example: `NameStore` in the glossary convention
- **UI suffix**: Crates ending in `_ui` provide UI components
  - Examples: `agent_ui`, `collab_ui`, `debugger_ui`, `settings_ui`
- **Test support**: Feature flag pattern `test-support` for testing utilities

## Build Configuration

The workspace `Cargo.toml` defines:

### Workspace Metadata

```toml
[workspace.package]
publish = false
edition = "2024"
```

### Optimization Profiles

- **Dev profile**: Split debuginfo, incremental compilation, 16 codegen units
- **Release profile**: Thin LTO, 1 codegen unit (except zed binary: 16)
- **Release-fast profile**: Full debug info, no LTO, 16 codegen units

### Proc-macro Optimization

Proc-macro crates are optimized even in debug builds:

- `gpui_macros`, `derive_refineable`, `settings_macros`, `sqlez_macros`,
  `ui_macros`, `util_macros`
- Built with `opt-level = 3` and `codegen-units = 1`
